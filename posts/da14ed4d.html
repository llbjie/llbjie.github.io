论文「The UNIX Time-Sharing System」翻译及阅读笔记

ABSTRACT 摘要

UNIX 是一个通用的，多用户的，交互性的操作系统，适用于”larger Digital Equipment Corporation，美国数字设备公司”PDP-11和 “Interdata” 8/32电脑。它提供了一些即使是在大型操作系统上也很少发现的特性，包括：

  1. 一种包含可拆卸卷的分层文件系统
  2. 兼容文件、设备和进程间I/O
  3. 具备启动异步进程的能力
  4. 可基于每位用户选择的系统命令语言
  5. 包含超过100个子系统，涵盖几十种编程语言
  6. 高移植性

这篇文章讨论了这些特性、文件系统、用户命令接口的实现

INTRODUCTION 引言

现在（指论文发表的时间）有四种版本的 UNIX 分式系统：

  1. 跑在 PDP-7 和 PDP-9 计算机
  2. PDP-11/20 计算机，无内存保护机制
  3. 多道程序设计, 可运行于PDP-11/34、/40、/45、/70型计算机
  4. PDP-11/70、Interdata 8/32计算机,也就是本文描述的系统

HARDWARE AND SOFTWARE ENVIRONMENT 软件和硬件环境

安装研究用 UNIX 系统的PDP-11/70是一台16位字长（8位字节）计算机，配备768字节核心内存；系统内核约占90字节，其中代码和数据表各占约一半空间。然而，本系统包含大量设备驱动程序，并为I/O缓冲区和系统表分配了充裕空间；能够运行前述软件的最小化系统，整体仅需96K字节核心内存。实际也存在规模更大的安装配置，例如PWB/UNIX系统（见文献45）。同时亦有更精简的系统版本，尽管功能存在一定限制（参见文献6）。

THE FILE SYSTEM 文件系统

操作系统最重要的功能是提供一个文件系统，从用户的角度来说，有三种文件：

  1. 普通磁盘文件
  2. 目录
  3. 特殊文件

普通文件

文件可包含用户存储的任何信息，例如符号程序或二进制（目标）程序。
系统不对其内部结构作特定预设。文本文件仅由字符串构成，各行之间以换行符分隔。二进制程序则是按程序执行时在核心内存中的呈现形式排列的字符序列。
文件的结构始终由使用它的程序控制，而非由操作系统本身规定。

目录

目录实现了文件名与文件实体之间的映射关系，从而构建了文件系统的整体架构。
目录在功能上完全等同于普通文件，唯一区别在于非特权程序无法对其执行写操作——这意味着系统始终掌控着目录的内容。但任何获得适当权限的用户都能像读取普通文件一样读取目录信息。

系统维护着若干个供其自身使用的目录

  1. 根目录（root directory， 这里指的是 ‘/‘目录）。系统中的所有文件均可通过沿着一连串目录追踪路径直至找到目标文件的方式来定位，此类搜索通常以根目录为起点。
  2. 其他系统目录则存放了所有供通用目的使用的程序，即所有的命令（这里指的是/bin目录）。然而程序并非必须驻留在这些目录中才能被执行。

检索文件的方式：

  1. 绝对路径，从’/‘开始，从根目录开始搜索文件
  2. 相对路径，不以’/‘开始，表示从当前目录开始搜索文件

同一个非目录文件可以在多个目录下出现，且可能使用不同的名称。这一特性称为链接；
UNIX系统与其他允许链接的系统不同之处在于，文件的所有链接都具有同等地位。也就是说，文件并非存在于某个特定目录之内；文件的目录项仅包含其名称以及一个指向实际描述该文件信息的指针。因此，文件独立于任何目录项而存在，尽管在实际操作中，当文件的最后一个链接被删除时，该文件也会随之消失。

每个目录至少包含两个条目。

  1. “.” 总是指向该目录自身。
  2. “..” 指向当前目录的父目录。

除特殊条目”.”和”..”外，每个目录必须且仅能作为一项出现在另一个目录（其父目录）中。
这样设计的原因是为了简化遍历目录结构子树的程序编写，更重要的是为了避免层次结构的部分脱离。如果允许对目录进行任意链接，将很难检测根目录到某个目录的最后一条连接何时被切断。

特殊文件

UNIX中的每个支持的I/O设备至少被关联一个特殊文件，特殊文件和普通文件的读写操作完全相同

特殊文件的目录项位于 /dev 目录中。

为什么这样处理I/O设备：

  1. 文件与设备的I/O操作尽可能相似
  2. 文件与设备名称具有相同的语法和含义，因此原本接收文件名作为参数的程序也可以直接传入设备名
  3. 特殊文件与普通文件受相同的保护机制约束

可拆卸的文件系统

尽管文件系统的根目录始终存储在同一设备上，但整个文件系统乘车并不全部驻留于此设备。系统提供了带两个参数的系统调用 mount：

  1. 普通文件的名称
  2. 特殊文件的名称，该特殊文件所关联的存储卷（如磁盘组）应具有一个包含自身目录层次结构的独立文件系统

这个系统调用的作用是，使原本指向该普通文件的引用转而指向可拆卸卷上文件系统的根目录。实际上，mount系统调用是用一颗全新的子树替换了层次树的叶节点（即那个普通文件）

对不同设备上的文件采取相同处理的原则有一个例外：不允许在不同文件系统层次结构之间创建链接。强制执行此限制是为了避免繁琐的记录工作，否则在卸载可拆卸卷时需要确保移除这些链接

保护

系统中每个用户被分配了一个唯一用户标识号（unique user identification number）。
创建文件时，会表示文件所有者的用户ID,同时，新文件还会获得一组共十个保护位。其中九位分别独立指定 文件所有者、所属组内其他成员以及所有其他用户对该文件的读、写和执行权限。

若第十位置位，则当该文件作为程序执行时，系统会将当前用户ID临时更改为文件创建者的用户ID。

这种用户ID的变更仅在执行调用该功能的程序期间有效。设置用户ID特性使得特权程序能够访问其他用户无法访问的文件。

例如，某个程序可能维护一个记账文件，该文件除了程序自身外不应被读取或修改。如果该程序的设置用户ID位被置位，它就能访问此文件，即使该程序用户调用的其他程序可能被禁止此访问。由于任何程序调用者的实际用户ID始终可被获取，设置用户ID的程序可以采取任何所需措施来验证调用者的凭据。此机制用于允许用户执行经过精心编写的命令，这些命令可调用特权系统入口。

例如，存在一个仅可由“超级用户”（见下文）调用的系统入口，用于创建空目录。如上所述，目录应包含“.”和“..”条目。创建目录的命令归超级用户所有，并且其设置用户ID位被置位。在验证调用者具有创建指定目录的授权后，它将创建目录并建立“.”和“..”条目。

由于任何用户都可以在自己的文件上设置用户ID位，此机制通常无需管理员干预即可使用。例如，此保护方案轻松解决了“Aleph-null”提出的MOO记账问题。

系统将一个特定用户ID（即“超级用户”的ID）识别为不受常规文件访问限制的特例；因此，可以编写程序来转储和恢复文件系统，而不会受到保护系统的意外干扰。

I/O 调用

系统调用执行I/O的设计旨在消除不同设备和访问方式之间的差异。系统中不存在“随机”与“顺序”I/O的区别，也没有强制规定任何逻辑记录大小。普通文件的大小由写入其中的字节数决定，无需也无法预先确定文件的大小。

为阐明I/O的基本原理，以下以一种匿名语言简要总结部分基础系统调用，该方式可体现必要参数而避免涉及底层复杂性。请注意，每次系统调用都可能返回错误，为简洁起见，调用序列中未体现错误处理机制。

文件系统不维护用户可见的锁，也不限制可同时打开同一文件进行读写的用户数量（但是用户可打开的文件数量是有限的）。
但是，系统内部设有充分的互锁机制，确保当两个用户同时进行如写入同一文件、在同一目录创建文件或删除彼此已打开文件等操作时，文件系统的逻辑一致性得以维持。

有下面几种系统调用：

  • 打开文件，open
  • 创建一个新文件或重写旧文件，create
  • 读/写文件，read、write
  • 随机访问，lseek

IMPLEMENTATION OF THE FILE SYSTEM 文件系统的实现

i-node：存储文件的信息

目录项仅包含关联文件的名称和一个指向文件本身的指针。该指针是整数，称为文件的i-number(索引编号)。当访问文件时，i-number被作为系统表（即i-list）的索引，该表存储在目录所在设备的已知区域。通过索引找到的条目（即该文件的i-node, 索引节点）包含了对该文件的描述

  • 文件所有者的用户ID和组ID
  • 保护位（访问权限）
  • 文件内容的物理磁盘或磁带地址
  • 文件大小
  • 创建时间、最后访问时间、最后修改时间
  • 文件的链接数，即该文件在目录中出现的次数
  • 标识文件类型的代码：目录、普通文件或特殊文件

文件操作与 i-node 机制

open或create系统调用的目的是通过搜索显式或隐式指定的目录，将用户提供的路径名转换为 i-node编号（索引节点编号）。

文件一旦被打开，其设备号、i-node编号及读/写指针会被存储在系统表中，该表通过调用返回的文件描述符进行索引。因此，在后续调用 read 或 write 时，可以轻松地通过文件描述符获取访问文件所需的信息。

当创建一个新文件时，系统会为其分配一个 i-node，并创建一个包含文件名和 i-node 编号的目录项（directory entry）。为现有文件创建链接（link）涉及创建一个具有新名称的目录项，复制原始文件的 i-node 编号，并递增该 i-node 的链接计数（link-count）字段。删除文件是通过递减其目录项指定的 i-node 链接计数并抹除该目录项来完成的。如果链接计数降至 0，则该文件占用的所有磁盘块都会被释放，且该 i-node 会被重新分配（注：即撤销分配）。

寻址架构与文件大小

包含文件系统的所有磁盘空间都被划分为若干个 512 字节的块，逻辑地址从 0 开始，上限取决于设备容量。每个文件的 i-node 空间可容纳 13 个设备地址。对于普通文件，

  • 前 10 个地址直接指向文件的前 10 个块。
  • 如果文件超过 10 块，第 11 个地址指向一个一级间接块（indirect block），该块包含多达 128 个指向文件后续块的地址。
  • 更大的文件使用第 12 个地址指向一个二级间接块，该块指向 128 个一级间接块。
  • 如有需要，第 13 个地址则作为三级间接块。

据此，文件在概念上最大可达 $[(10 + 128 + 128^2 + 128^3) \times 512]$ 字节。文件打开后：读取 5120 字节以内的内容只需一次磁盘访问；5120 至 70,656 字节范围需两次访问；70,656 至 8,459,264 字节范围需三次访问；从此范围到最大文件容量（1,082,201,088 字节）则需四次访问。
在实践中，设备缓存机制能有效消除大部分间接读取开销。

实际上，设备缓存机制（见下文）能有效消除大部分间接寻址访问。

特殊文件与挂载（Mount）

以上讨论适用于普通文件。当对某个文件发起I/O请求，而其i-node指示其为特殊文件时，最后12个设备地址字无意义，第一个地址字指定一个内部设备名，该名称被解析为一对数字，分别表示设备类型和子设备号。设备类型决定由哪个系统例程处理该设备的I/O；子设备号则用于选择具体设备，例如连接至特定控制器的某个磁盘驱动器，或若干相似终端接口中的某一个。

在这种环境下，mount 系统调用（第3.4节）的实现相当直接。mount 维护了一个系统表，其键是挂载时指定的普通文件的 i-number 和设备名，对应的值是指定的特殊文件的设备名。在 open 或 create 操作中扫描路径名时，每当遇到一个 i-number/设备对，就会搜索此表；如果找到匹配项，该 i-number 会被替换为被挂载文件系统根目录的 i-number，设备名则被替换为表中的对应值。

缓存与 I/O 效率

对用户而言，文件的读写似乎是同步且无缓冲的：read 调用返回后数据立即生效，write 返回后用户空间即可重用。实际上，系统维护了一套相当复杂的缓冲机制。

写操作： 即使只写一个字节，系统也会检查受影响的磁盘块是否已在主存缓冲区中；若不在，则从设备读入。修改该字节后，该块被标记为待写入。此时 write 即可返回，实际的 I/O 可能会推迟完成。

读操作： 同理，读取单字节时若块已在缓冲区则立即返回；否则读入缓冲区后再提取。

系统能识别程序对文件的顺序访问，并异步地预读（pre-read）下一块，这显著降低了运行时间。以 512 字节为单位进行读写的程序比逐字节读写的程序更有优势，但这主要是因为减少了系统调用开销（System Overhead）。

i-list

i-list 的概念是 UNIX 的一个独有特点。实践证明，这种组织文件系统的方法非常可靠且易于管理。对系统本身而言，它的一个优势在于每个文件都有一个简短、明确的名称，并以简单的方式与访问该文件所需的保护、寻址等信息相关联。同时，它也允许采用一种相当简单且快速的算法来检查文件系统的一致性，例如验证设备上包含有用信息的区域与可被分配的空闲区域是互不相交的，且共同覆盖整个设备空间。该算法独立于目录层次结构，因为它只需要扫描线性组织的 i-list。与此同时，i-list 的概念也带来了一些其他文件系统组织方式中没有的特殊性。例如，应由谁来承担文件占用空间的费用，因为一个文件的所有目录条目都具有同等地位。向文件所有者收费通常并不公平，因为一个用户可能创建了文件，另一个用户可能链接到它，而第一个用户又可能删除该文件。第一个用户仍然是文件的所有者，但费用却应向第二个用户收取。最合理的公平算法似乎是将费用平均分摊给所有链接到该文件的用户。许多安装环境通过完全不收取任何费用来回避这个问题。

PROCESSES AND IMAGES 进程与映像

映像(Image)是一个计算机执行环境。它包含内存映像、通用寄存器值、打开文件的状态、当前目录等。映像是一个“伪计算机”的当前状态。

进程（Process）是映像的一次执行过程。当处理器代表某个进程执行时，其映像必须驻留在主存中；在其他进程执行期间，该映像仍保留在主存中，除非出现更高级别的活跃进程，迫使其被交换（Swap）到磁盘上。

映像的用户内存部分分为三个逻辑段：

  • 程序正文段（Program Text Segment）： 从虚拟地址空间的地址 0 开始。在执行期间，该段受写保护，且在所有执行同一程序的进程之间共享同一个副本。
  • 数据段（Data Segment）： 位于虚拟地址空间中程序正文段上方的第一个硬件保护字节边界处。该段是非共享且可写的，其大小可以通过系统调用进行扩展。
  • 栈段（Stack Segment）： 从虚拟地址空间的最高地址开始。随着栈指针的波动，该段会自动向下增长。

Processes 进程

出了系统正在引导(bootstrapping)自身运行的特殊情况外，新进程只能通过调用fork系统调用产生:

当fork执行时，进程分裂成两个独立执行的进程。这两个进程拥有原始内存映像的独立副本，并共享所有打开的文件。这两个新进程的唯一区别在于其中一个被视为父进程：在父进程中，返回的 processid 实际上标识了子进程且绝不为 0；而在子进程中，返回的值始终为 0。

由于 fork 在父进程和子进程中返回的值是可区分的，因此每个进程都可以判定自己是父进程还是子进程。

bootstrapping
是指计算机从断电状态（冷启动）或重置状态（热启动）到操作系统完全加载并运行的过程。硬件设计者在主板的非易失性存储器（如 ROM 或 Flash）中固化了一小段代码。这段代码必须极度精简，其唯一的使命就是：寻找并加载一个稍微大一点的程序，而那个程序再去加载操作系统。

操作系统的启动过程：

  1. 硬件自检查，执行存储在主板 ROM 中的 BIOS（基本输入输出系统）或 UEFI（统一可扩展固件接口）。识别可启动设备（硬盘，U盘，网络等）
  2. BIOS/UEFI 会读取可启动设备上起始位置的一个特殊扇区
    • MBR：传统方式下， BIOS会读取磁盘的第一个512字节块，包含一小段引导代码和分区表
    • EFI系统分区：在现代 UEFI 系统中，固件直接识别 FAT32 分区并运行其中的 .efi 文件
  3. 引导加载程序(bootloader),由于512字节空间太小，无法装下复杂的内核，因此 MBR 中的代码通常只负责加载 二级引导加载程序
    功能：提供启动菜单，让用户选择操作系统；加载内核镜像到内存中
  4. 内核初始化，内核接管了计算机的控制权，开始执行以下操作：
    • 建立页表：初始化内存管理单元(MMU)以支持虚拟内存
    • 驱动加载：识别并初始化核心硬件驱动程序
    • 挂载根文件系统：找到包含操作系统文件的磁盘分区
  5. 创建第一个进程
    • 内核在用户空间启动第一个进程。在类 Unix 系统中，这个进程通常被称为 init 或 systemd
    • 这个进程随后会派生 fork 出所有其它系统服务(如登录界面、网络服务、图形界面)

总结 ：BIOS 找到 引导加载程序，引导程序负责将 内核 装入内存并启动；内核随后通过 挂载根文件系统 唤醒整个 操作系统

Pipes 管道

进程可以使用与文件系统I/O相同的系统调用 (read和write)与相关进程进行通信。调用：

对于大多数用户而言，与系统的通信时借助一个名为 shell 的程序完成的。shell 是一个命令解释器：它读取用键入的行，并将其解释为执行其它程序的请求。

Standard I/O 标准输入输出

上文第三节对 I/O 的讨论似乎暗示，程序所使用的每一个文件都必须由该程序亲自打开或创建，才能获得该文件的文件描述符。然而，由 Shell 执行的程序在启动时就已经拥有三个打开的文件，其文件描述符分别为 0、1 和 2。

Implementation of the shell Shell 的实现

现在可以理解 Shell 运行的梗概了。大部分时间，Shell 都在等待用户键入命令。当结束该行的换行符被键入时，Shell 的 read 调用返回。Shell 分析命令行，并将参数整理成适合 execute 的形式。随后调用 fork。子进程（其代码当然仍是 Shell 的代码）尝试使用相应的参数执行 execute。如果成功，这将读入并开始执行所给名称的程序。与此同时，由 fork 产生的另一个进程（即父进程）则等待子进程终止。当子进程终止时，Shell 知道命令已执行完毕，于是键入提示符并读取键盘以获取另一个命令。

在此框架下，后台进程的实现变得易如反掌：每当命令行包含 “&” 时，Shell 仅仅只需不对它为执行命令而创建的进程执行等待（wait）即可。

令人欣慰的是，所有这些机制都与标准输入输出文件的概念契合得非常完美。当通过 fork 原语创建一个进程时，它不仅继承了父进程的内存映像，还继承了父进程中当前所有打开的文件，包括文件描述符为 0、1 和 2 的文件。当然，Shell 使用这些文件来读取命令行并写入提示符和诊断信息，而在通常情况下，它的子进程——即那些命令程序——会自动继承它们。然而，当给出带有“<”或“>”的参数时，后代进程在执行 execute 之前，会使标准 I/O 文件描述符（分别为 0 或 1）指向命名的文件。这很容易实现，因为根据约定，当打开（或创建）一个新文件时，会分配最小的未使用文件描述符；因此只需关闭文件 0（或 1）并打开命名的文件即可。由于运行命令程序的进程在完成后会直接终止，因此在“<”或“>”之后指定的文件与文件描述符 0 或 1 之间的关联会在进程死亡时自动结束。因此，Shell 无需知道其自身标准输入输出文件的实际名称，因为它永远不需要重新打开它们。

过滤器是标准 I/O 重定向的直接扩展，只是使用了管道（pipe）代替文件。

在通常情况下，Shell 的主循环永远不会终止。（主循环包括 fork 返回中属于父进程的分支；即执行 wait 随后读取另一个命令行的分支。）导致 Shell 终止的唯一情况是在其输入文件中发现了文件结束（EOF）条件。因此，当 Shell 作为一个带有给定输入文件的命令执行时，如：

Initialization 初始化

用户在其上键入命令的 Shell 实例，其本身也是另一个进程的子进程。系统初始化的最后一步是创建一个单一进程，并（通过 execute）调用一个名为 init 的程序。init 的角色是为每个终端通道创建一个进程。init 的各个子实例分别为文件 0、1 和 2 打开相应的终端以进行输入和输出；如有必要，还会在拨号线路上等待载波建立。随后，系统会打印出一条消息，请求用户登录（log in）。

当用户键入姓名或其他身份标识时，相应的 init 实例会被唤醒，接收登录行并读取密码文件。如果找到了该用户名，且用户能够提供正确的密码，init 就会切换到该用户的默认当前目录，将进程的用户 ID（User ID）设置为该登录者的 ID，并执行 execute 来启动 Shell。至此，Shell 已准备好接收命令，登录协议执行完毕。

与此同时，init 的主干路径（即后来演变为 Shell 的所有子实例的父进程）会执行 wait 调用。如果其中一个子进程终止——无论是由于 Shell 发现了文件结束符（EOF），还是由于用户键入了错误的姓名或密码——init 的这条主干路径只需简单地重新创建那个已失效的进程，而该进程又会重新打开相应的输入输出文件并打印另一条登录消息。因此，用户只需向 Shell 键入文件结束序列即可实现注销（log out）。

Other programs as shell 其他程序作为 Shell

上文所述的 Shell 旨在允许用户完全访问系统的各项设施，因为它能够调用并执行任何具有相应保护模式（权限）的程序。然而，有时我们需要一种不同的系统交互界面，而这一特性很容易实现。

回想一下，在用户通过提供姓名和密码成功登录后，init 通常会调用 Shell 来解释命令行。用户在密码文件（password file）中的条目可以包含登录后要调用的特定程序名称，以此替代标准 Shell。该程序可以按照其意愿自由地解释用户的消息。

例如，对于使用秘书编辑系统的用户，其密码文件条目可能会指定使用编辑器 ed 来替代 Shell。这样，当该系统的用户登录时，他们直接处于编辑器内部并可以立即开始工作；同时，这也可以防止他们调用非预期的程序。在实践中，允许从编辑器中“临时逃逸”（temporary escape）以执行格式化程序和其他实用工具已被证明是切实需要的。

系统上提供的几种游戏（如象棋、二十一点、三维井字棋）展示了一种受到更严格限制的环境。对于这些游戏，密码文件中都存在一个条目，指定调用相应的游戏程序而非 Shell。以这些游戏玩家身份登录的人会发现自己被限制在游戏之内，无法探索 UNIX 系统整体提供的（推测可能更有趣的）功能。

TRAPS 陷阱

DP-11 硬件可以检测到许多程序错误，例如引用不存在的内存、执行未实现的指令，以及在需要偶数地址的地方使用了奇数地址。此类错误会导致处理器“陷阱”（trap）到某个系统例程中。除非另有安排，否则非法操作会导致系统终止该进程，并将其映像（image）写入当前目录下的 core 文件中。调试器可用于确定发生故障时程序的执行状态。

对于那些陷入死循环、产生多余输出或使用户改变主意的程序，可以通过使用中断信号（interrupt signal）将其停止，该信号通过键入“delete”字符生成。除非采取了特殊措施，否则该信号仅会导致程序停止执行，而不产生 core 文件。此外，还有一个退出信号（quit signal）用于强制生成映像文件。因此，非预期循环的程序可以被停止，且无需预先安排即可检查其遗留状态。

硬件生成的故障以及中断和退出信号，均可根据请求被进程忽略或捕获。例如，Shell 会忽略退出信号（quit），以防止退出信号导致用户注销。编辑器会捕获中断信号并返回到其命令级别。这对于在不丢失正在进行的工作（编辑器操作的是其所编辑文件的副本）的情况下停止长内容的打印输出非常有用。在没有浮点硬件的系统中，未实现的指令会被捕获，并对浮点指令进行软件模拟解释。

PERSPECTIVE 展望

或许有些悖论的是，UNIX 系统的成功在很大程度上归功于它并非为了满足任何预定义的目标而设计。当作者之一（Thompson）对当时的计算机设施感到不满，发现了一台极少被使用的 PDP-7，并着手创建一个更舒适（hospitable）的环境时，第一个版本便诞生了。这一（本质上是个人性质的）努力取得了足够的成功，吸引了另一位作者及几位同事的兴趣，并随后促成了对 PDP-11/20 的收购，专门用于支持文本编辑和格式化系统。当 11/20 也不堪重负时，该系统已证明了其价值，足以说服管理层投资购置 PDP-11/45，以及后来的 PDP-11/70 和 Interdata 8/32 机器，系统在这些机器上发展成了现在的形式。在整个过程中，我们的目标（如果非要阐明的话）始终是建立一种与机器之间舒适的关系，并探索操作系统及其他软件中的理念与发明。我们未曾面临满足他人需求的压力，对于这种自由，我们深表感激。

回顾过去，有三点考量影响了 UNIX 的设计：

  • 首先：因为我们是程序员，所以很自然地将系统设计得易于编写、测试和运行程序。 我们追求编程便利性的最重要的表现是系统被安排为交互式使用，尽管原始版本仅支持单用户。我们相信，一个设计良好的交互式系统比“批处理”系统更高效、使用起来更令人愉悦。此外，此类系统很容易适配非交互式用途，而反之则不然。

  • 其次：系统及其软件始终受到相当严格的尺寸限制。 鉴于在合理的效率与表达能力之间存在部分对立的欲望，尺寸限制不仅鼓励了精简，还促成了某种设计的优雅。这可能是“苦难成就救赎”哲学的一个隐蔽版本，但在我们的案例中，它奏效了。

  • 第三：几乎从一开始，系统就能做到并确实在进行“自我维护”。 这一事实比表面看起来更重要。如果系统的设计者被迫使用该系统，他们会迅速察觉到其功能和表面的缺陷，并有强烈的动力在为时已晚之前纠正它们。由于所有的源程序始终可用且易于在线修改，当新想法被发明、发现或由他人建议时，我们愿意修订并重写系统及其软件。

本文讨论的 UNIX 各个方面清晰地展示了至少前两项设计考量。例如，从编程角度来看，文件系统的接口极其便利。最低限度的接口层旨在消除各种设备与文件之间、以及直接访问与顺序访问之间的区别。程序员无需庞大的“访问方法”例程来与系统调用隔离开；事实上，所有用户程序要么直接调用系统，要么使用一个不到一页纸长的微型库程序，该程序负责缓冲若干字符并一次性读写。

编程便利性的另一个重要方面是不存在复杂的“控制块”结构（即文件系统或其他系统调用所依赖或维护的部分结构）。通常来说，程序地址空间的内容属于程序所有，我们尽量避免对该地址空间内的数据结构设置限制。

鉴于所有程序都应能使用任何文件或设备作为输入或输出的要求，将设备相关性推入操作系统内部也是明智的。唯一的替代方案要么是在所有程序中加载处理每个设备的例程（这会消耗大量空间），要么依赖某种在实际需要时动态链接到对应设备例程的手段（这在系统开销或硬件上都很昂贵）。

同样，进程控制方案和命令行接口已被证明既方便又高效。由于 Shell 作为一个普通的、可交换（swappable）的用户程序运行，它在系统内核中不占用任何“常驻”空间，并且可以以极低的成本实现极其强大的功能。特别是在 Shell 作为一个产生其他进程来执行命令的进程框架下，I/O 重定向、后台进程、命令文件以及用户可选的系统接口这些概念在实现上都变得本质上非常简单（trivial）。

INFLUENCES 灵感来源

UNIX 的成功与其说在于全新的发明，不如说在于对一组精心挑选的、富有生命力的思想进行了充分挖掘。特别地，它证明了这些思想是实现一个虽小巧却强大的操作系统的关键。

我们所实现的 fork 操作，其本质在 GENIE 分时系统中就已经存在。在许多方面，我们受到了 Multics 的影响：它启发了 I/O 系统调用的特定形式，以及 “Shell” 的名称及其通用功能。Shell 应该为每个命令创建一个进程的理念，也是由 Multics 的早期设计启发而来的，尽管在那个系统中，后来出于效率原因放弃了这一做法。类似的方案在 TENEX 系统中也有所应用。

Operating system interfaces 操作系统接口

操作系统的任务是在多个程序之间共享计算机资源，并提供比单纯硬件支持更有用的服务。具体来说有这下面几个功能：

  1. 管理和抽象正在使用的硬件。例如应用程序并不需要关心正在使用的磁盘类型。
  2. 在多个程序之间共享资源，以至应用程序（看起来）能够在同一时间运行
  3. 操作系统为程序交互提供了受控的方式，使它们能够共享数据或协同工作。

操作系统通过接口 interface 向用户程序提供功能，如何设计这些接口呢？

  • 一方面我们希望接口简洁而精炼，因为这有助于更准确地实现功能。
  • 另一方面，我们又倾向于为应用程序提供许多复杂高级的功能。

解决这一矛盾的关键在于：设计基于少量核心机制的接口，这些机制能够灵活组合，从而提供广泛的通用性。

Processes and memory 进程和内存

一个 xv6 进程由用户空间内存（包括指令、数据和栈）以及内核维护的进程私有状态组成。xv6 采用分时共享机制管理进程：它在等待执行的进程集合中透明地切换可用的 CPU 资源。当某个进程暂时不执行时，xv6 会保存其 CPU 寄存器中的状态，并在下次调度运行该进程时恢复这些状态。内核为每个进程关联一个唯一的进程标识符，即 PID。

I/O 和文件描述符 I/O and File descriptors

在内部，xv6 内核使用文件描述符作为每个进程表的索引，因此每个进程都有一个从 0 开始的私有文件描述符空间。

按照惯例，进程从文件描述符 0（标准输入）读取数据，将输出写入文件描述符 1（标准输出），并将错误消息写入文件描述符 2（标准错误）。

fork 会复制父进程的文件描述符表及其内存，因此子进程启动时拥有与父进程完全相同的打开文件。exec系统调用会替换调用进程的地址空间（内存内容），但会保留其文件描述符表。这一行为使得 Shell 能够通过以下方式实现 I/O 重定向：首先 fork一个子进程，然后在子进程中重新打开选定的文件描述符，最后调用 exec 来执行新的程序。

Pipes 管道

管道（pipe）是一个小的内核缓冲区，以一对文件描述符的形式暴露给进程：一个用于读取，另一个用于写入。

操作系统实现 cmd1 | cmd2的方式：

  • 首先 fork 出一个子进程执行命令，分析命令的格式
  • 如果包含 |, | 的两边都会fork出一个进程，然后 exec 对应的命令，然后等待两边的进程都结束
  • 如果 cmd1 同样包含 |, 它也会 fork 出两个子进程然后调用 exec。最终会形成一棵进程树,叶节点是需要执行的命令，中间结点是等待两侧结束的进程

Operating system organization 操作系统组织

一个操作系统必须满足三个要求：

  • 多路复用, 指的是操作系统（看起来）能同时运行超过硬件CPU数目的应用程序，
  • 隔离，每个进程都感觉自己独占了计算机资源，不会影响到其它进程
  • 交互，提供进程之间交互的功能，如 pipe

本章概述了操作系统是如何组织起来以实现这三个要求的。实现这些目标的方法有很多，但本书主要关注以单体内核为核心的主流设计，这种设计被许多 Unix 操作系统所采用。本章还介绍了 xv6 中的进程——它是 xv6 中实现隔离的基本单位——以及 xv6 启动时第一个进程的创建过程。

什么是单体内核:这是一种操作系统内核的设计架构，即所有的功能都运行在内核态

xv6 运行在一个多核（multi-core）RISC-V 微处理器上，其许多底层功能（例如进程的实现）都是针对 RISC-V 架构特定设计的。RISC-V 是一种 64 位 CPU，而 xv6 使用的是“LP64”模型的 C 语言编写，这意味着在 C 语言中，long（L）和指针（P）是 64 位的，而 int 是 32 位。
在一台完整的计算机中，CPU 周围配备了各种支持硬件，其中大部分以 I/O 接口的形式存在。xv6 是为 qemu 的 “-machine virt” 选项所模拟的支持硬件编写的。这些硬件包括：RAM、一段包含引导代码的 ROM、用于连接用户键盘和屏幕的串行接口，以及用于存储数据的磁盘设备。

  1. 前面提到了 单体内核，这里提到了多核，这两者的区别和联系是什么：这里的多核是指一个芯片里有多少个计算核心，属于硬件级别的概念。
  2. 什么是 qemu,一个硬件模拟器，模拟了RAM,虚拟磁盘，串口等
  3. qemu和虚拟机，和windows的linux子系统，这三者的功能有些相似，解释它们之间的区别。 qemu 完全使用软件模仿了硬件，可以跨架构运行；虚拟机不模拟cpu等硬件，而是共享真实硬件，只虚拟内存，I/O,不能跨架构运行；wsl2使用了 hyper-v虚拟机，这里虚拟机比较轻量，然后运行linux内核，所以还是需要下载一个linux内核

Abstracting physical resources 抽象物理资源

为什么需要操作系统？我们完全可以将一些需要用到的系统调用打包成一个库，然后在程序编译的时候链接上这个库，甚至每个应用程序都可以有一个自己特殊定制的库，可以直接和硬件交互，以自身最合适的方式使用这些资源。

‘DB-OS’协同设计：硬件发展的速度，已经远远甩开了操作系统通用接口的演进速度。以往CPU花费在系统调用上的开销很小，而现在往往是CPU在等待操作系统。操作系统本身正在成为新的“慢速磁盘”。像数据库这种软件，权限比一般的应用更大，在某些情况下也应该有操作系统级别的权限。如何进行“DB-OS协同设计”正在成为一个至关重要的问题。

User mode, supervisor mode, and system calls 用户态、监督态与系统调用

为了实现强隔离，操作系统必须确保：应用程序不能修改（甚至不能读取）操作系统的数据结构和指令，并且应用程序也不能访问其他进程的内存。CPUs提供了硬件上的支持。
例如 RISC-V有三种cpu执行指令的模式：

  • 机器态，运行在此种状态的指令有完全的特权，CPU 在启动时最初处于机器态。
  • 监督态，xv6 只在机器态下执行少量代码，然后就切换到监督态运行。允许执行特权指令。
  • 用户态

在监督态下，cpu被允许执行特权指令，如

  • 启用或关闭终端
  • 读写保存页表地址的寄存器

如果一个运行在用户态的应用程序试图执行特权指令，CPU 不会执行该指令，而是会切换到监督态，由监督态代码来处理这种情况——通常会终止该应用程序，因为它执行了不被允许的操作

用户空间和内核空间
应用程序只能执行用户态指令（例如加法运算等），此时称其运行在用户空间（user space）；而运行在监督态的软件不仅可以执行普通指令，还可以执行特权指令，此时称其运行在内核空间（kernel space）。

CPU 提供了一种特殊指令，可以将 CPU 从用户态切换到监督态，并从内核指定的入口点进入内核。（在 RISC-V 中，这个指令是 ecall。）
当 CPU 切换到监督态之后，内核就可以对系统调用的参数进行验证（例如，检查传入的地址是否属于该应用程序的内存空间），并判断该应用是否有权限执行请求的操作（例如，检查该应用是否有权限写入指定文件），然后决定拒绝还是执行该操作。

Kernel organization 内核组织

一个关键设计问题是 操作系统的那些部分应该允许在监督态。有两种设计架构：

  • 宏内核，整个操作系统都运行在监督态中，所有的功能都拥有完整特权，所以操作系统的组织设计十分方便；此外操作系统的各个部分可以很容易进行协同。这种设计的缺点在于:操作系统各个组成部分之间的接口往往十分复杂，因此系统开发者很容易出错，而且发生在监督态的错误通常会导致内核崩溃。
  • 微内核，只有必要的功能运行在内核态，其余功能放在用户态。内核提供进程间的通信机制。
  1. 在微内核架构中，那些功能应该放在用户态，哪些功能应该放到监督态
  2. 这两种架构各自的应用场景是什么

从本书的角度来看，微内核和单体内核操作系统共享许多关键思想。它们都会实现

  • 系统调用
  • 使用页表
  • 处理中断
  • 支持进程
  • 使用锁进行并发控制
  • 实现文件系统等

本书主要关注这些核心思想。

Process overview 进程概览

xv6（包括其它 unix 系统）中隔离的基本单位是 进程
内核用于实现进程的机制包括：

  • 用户态/内核态标志（user/supervisor mode flag）
  • 地址空间（address spaces）
  • 以及线程的时间片轮转（time-slicing）。

xv6 使用页表（由硬件实现）为每个进程提供各自独立的地址空间。RISC-V 的页表会将虚拟地址（即 RISC-V 指令所操作的地址）转换（或“映射”）为物理地址（即 CPU 芯片发送到主存的地址）。

xv6 为每个进程维护一个独立的页表，用来定义该进程的地址空间。

有多个因素限制了进程地址空间的最大大小：RISC-V 中的指针是 64 位宽；硬件在通过页表查找虚拟地址时只使用低 39 位；而 xv6 在这 39 位中只使用了 38 位。因此，最大地址为 2^38 − 1 = 0x3fffffffff，这个值被称为 MAXVA（定义于 kernel/riscv.h:363）。

在地址空间的顶部，xv6 预留了一个页面用于 trampoline（跳板），以及一个页面用于映射进程的 trapframe。xv6 利用这两个页面在用户态与内核态之间进行切换：

  • trampoline 页面包含进入和退出内核的代码，
  • 而映射 trapframe 则是为了保存和恢复用户进程的状态，这一点将在第 4 章中详细说明。

进程的相关信息被保存到 struct proc中，其定义位于 kernel/proc.h：

最重要的信息是

  • page table, 页表
  • kstack , 内核栈
  • state , 进程状态,例如：已分配（allocated）、可运行（ready to run）、正在运行（running）、等待 I/O（waiting for I/O）或已退出（exiting）

每个进程都有两个栈：用户栈和内核栈（p->kstack）。当进程执行用户态指令时，只使用用户栈，此时内核栈是空的。当进程进入内核（例如进行系统调用或响应中断）时，内核代码会在该进程的内核栈上执行；此时，用户栈中仍然保存着数据，但不会被主动使用。
因此，一个进程的线程会在用户栈和内核栈之间交替使用。内核栈是独立的（并且对用户代码是受保护的），这样即使某个进程破坏了自己的用户栈，内核仍然可以正常执行。

一个进程可以通过执行 RISC-V 的 ecall 指令来发起系统调用。该指令会提升硬件特权级别，并将程序计数器（program counter）切换到内核定义的入口点。

入口点处的代码会切换到内核栈，并执行实现该系统调用的内核指令。当系统调用完成后，内核会切换回用户栈，并通过执行 sret 指令返回用户空间。sret 会降低硬件特权级别，并从系统调用指令之后的位置继续执行用户程序。

进程的线程也可以在内核中“阻塞”（block），以等待 I/O 操作完成，并在 I/O 完成后从原来的位置继续执行。

总结发起系统调用的过程

  1. 用户程序触发 trap(ecall/中断/异常)
  2. cpu 进入 trampoline，执行 trampoline.S代码
    1. 保存用户寄存器到 trapframe
    2. 切换页表：user -> kernel
    3. 切换栈：用户栈 –> 内核栈
    4. 跳转到内核函数
  3. 在内核栈上运行内核代码
  4. 内核调用 usertrapret(),回到 trampoline
  5. trampoline 恢复现场
  6. 执行 sret,
  7. 回到用户态，用户程序继续执行

  1. 当 RISC-V 计算机上电时，它会进行初始化，并运行一个存储在只读存储器中的引导加载程序（boot loader）。该引导加载程序会将 xv6 内核加载到内存中。随后，在机器模式（machine mode）下，CPU 从 _entry（kernel/entry.S:7）开始执行 xv6。RISC-V 在启动时页式内存管理硬件是关闭的：虚拟地址会直接映射到物理地址。

  2. kernel/start.c 在 machine mode 下完成必要的 CPU 初始化（设置特权返回级别、关闭分页、配置中断与内存访问权限、设置定时器与 hartid），然后通过 mret 伪造返回到 S-Mode 并跳转到 main()。

  3. kernel/main.c 函数：在 S-Mode 下初始化内核页表、进程池、驱动程序等软件子系统，并最终启动第一个用户进程(调用userinit)以进入系统运行状态：

    • forkret 调用，在这时会加载并执行 /init 程序

  4. user/init.c，创建一个新的console文件（没有时），打开文件描述符 0、1、2

Security Model 安全模型

操作系统必须假设：一个进程中的用户级代码会尽一切可能破坏内核或其他进程。

内核的目标是限制每个用户进程，使其只能执行以下操作：读/写/执行其自身的用户内存、使用 32 个通用 RISC-V 寄存器，并且仅通过系统调用所允许的方式影响内核和其他进程。内核必须阻止任何其他行为。这通常是内核设计中的绝对要求。

对于内核自身代码的假设则完全不同。我们认为内核代码是由善意且谨慎的程序员编写的，应该是没有错误的，更不应包含恶意行为。

Real world

大多数操作系统都采用了“进程（process）”这一概念，而且大多数进程的结构与 xv6 中的进程类似。不过，现代操作系统通常支持在一个进程内包含多个线程（threads），以便单个进程能够利用多个 CPU。

在一个进程中支持多线程需要相当复杂的机制，而这些是 xv6 所不具备的，其中还包括对接口的改动（例如 Linux 中的 clone，它是 fork 的一种变体），用于控制线程之间共享进程哪些资源。

Page tables 页表

paging hardware 分页硬件

疑问：页面映射是由硬件完成的吗？进程数据结构中的 proc-pagetable 起到什么作用？
答案是：地址翻译（Address Translation）由硬件完成，但页表的构建和维护由软件（操作系统）完成。

xv6 使用 SV39模式，即64位虚拟地址中只有最低39位被使用。其中高27位用于 page table的索引, 低12位 是页内偏移(offset)。
每个 PTEs 的高44位被用于物理页号（physical page number, PPN），其余是一些标志位(flags)
PPN 和 offset 拼接最后形成了完整的物理地址：PA = concat(PPN, offset)

  • 这里可以推出 page table 最多可以有 2^27 个元素（页表项，page table entries, PTEs）
  • 也可以推出，一个页的大小是 2^12B = 4096字节
  • 每个 PTEs具体多少位不确定，取决于系统地址的长度，文中的长度为56，可以推出这里的标志位为12位
  • 虚拟地址的长度 = 39， 物理地址长度等于 56， 两者不相等

RISC-V的页表以三级树的形式存储在物理内存中。

从上图也可以看出 flags 中有哪些位

MMU将虚拟地址翻译成物理地址，总的流程如下：

  1. 从寄存器 satp 中取到最高级页表的物理地址，这一步不存在地址翻译
  2. 虚拟地址中的高9位作为中间级页表的索引，从而获得了中间级页表的物理地址
  3. 同理，虚拟地址中间9位作为最低级页表的索引，获得了最低级页表的物理地址
  4. 低9位作为最低级页表的索引获得了 PPN
  5. 拼接 PPN 和 offset 获得了最终的物理地址

多级页表相对于单级页表的优势在于能够很好的节约系统空间。

从上面的流程可以看到，如果需要翻译一个虚拟地址，需要加载三次内存，为了解决这个问题，引入了 Translation Look-aside Buffer (TLB)

澄清一些书中关键概念：

  • 物理内存（Physical memory）: RAM 中的存储单元
  • 物理地址（physical address）: 1字节的物理内存的地址
  • 地址空间（address space）：给定页表中有效虚拟地址的集合
  • 用户内存（user memory）: 进程的用户地址空间 + 进程页表允许访问的物理内存
  • 虚拟内存（Virtual memory）：与页表管理相关的概念和技术，以及利用页表实现诸如隔离性等目标的方法。

Process address space 进程地址空间

每个进程有自己的页表，当内核切换进程时，页表也会切换。进程的地址空间从0开始，MAXVA结束，但实际上只有小部分被映射到物理内存
进程的地址空间包含下面几个页：

  • 程序的文本（text）段, 权限为 PTE_R, PTE_X, PTE_U
  • 程序预先初始化的数据
  • 栈, PTE_R,PTE_W,PTE_U
  • 堆, PTE_R,PTE_W,PTE_U

在用户地址空间内使用权限是一种加固用户进程的常用技术。如果文本段被映射为 PTE_W，那么进程可能会意外修改自己的程序；xv6 在映射文本段时不带 PTE_W 权限；如果程序意外尝试向地址 0 执行存储操作，硬件将拒绝执行该存储操作并引发一个页面错误（参见第 4 章）。然后内核会终止该进程并打印出一条信息性消息，以便开发者能够追踪问题。

栈是一个单独的页面，图中显示了由 exec 系统调用创建的初始内容。包含命令行参数的字符串以及指向它们的指针数组位于栈的最顶部。在其下方是一些值，这些值允许程序从 main 开始执行，就好像函数 main(argc, argv) 刚刚被调用一样。

为了检测用户栈是否溢出了已分配的栈内存，xv6 通过清除 PTE_U 标志，在栈的正下方放置了一个不可访问的保护页。如果用户栈溢出，并且进程试图使用栈下方的地址，硬件将生成一个页面错误异常，因为在用户模式下运行的程序无法访问该保护页。而现实世界中的操作系统可能会在用户栈溢出时自动为其分配更多内存。

我们在此看到几个使用页表的很好的例子。首先，不同进程的页表将用户地址转换为不同的物理内存页面，因此每个进程都拥有私有的用户内存。其次，每个进程看到自己的内存具有从零开始的连续虚拟地址，而进程的物理内存则可以是非连续的。第三，内核在用户地址空间的顶部映射了一个包含跳板代码的页面（不带 PTE_U 标志），因此一个单独的物理内存页面出现在所有地址空间中，但只能由内核使用。

Real world 现实世界

  1. 分页与直接映射：xv6 在虚拟地址和物理地址之间使用简单的“直接映射”，并假设物理内存从固定地址 0x80000000 开始。这在 QEMU 模拟器中可行，但在真实硬件上很危险，因为真实硬件的内存布局不可预测。更成熟的内核会利用页表将杂乱的物理布局映射为规整的虚拟地址布局。
  2. 未使用的硬件特性：xv6 没有使用 RISC-V 提供的物理地址保护功能，也没有使用地址空间标识符（ASID）来优化 TLB 刷新（避免切换进程时刷新整个 TLB）。
  3. 页面大小的权衡：讨论了超级页面与大内存的适用关系。小内存用小页面（精细分配、方便换出），大内存用大页面（减少页表开销）。xv6 在这方面设计较简单。
  4. 内存分配器的局限：xv6 内核缺少类似 malloc 的小内存分配器，只能分配固定 4096 字节的块，这限制了它使用需要动态内存的复杂数据结构。真正的内核分配器需要灵活处理各种大小的请求。
  5. 设计倾向的变迁：内存分配是一个经典难题，过去更关注空间效率，而如今的设计趋势更看重速度。

Traps and system calls 陷阱和系统调用

有三种事件会导致 CPU 暂停常规的指令执行，并强制将控制权转移给处理该事件的特定内核代码。

  • 系统调用，即用户程序执行 ecall 指令来请求内核为其执行某项操作。
  • 异常：某条指令（用户态或内核态）执行了非法操作，例如从无效的虚拟地址加载数据。
  • 设备中断，即设备发出需要关注的信号，例如磁盘硬件完成了一个读写请求。

Xv6 的陷入处理分为四个阶段进行：

  1. RISC-V CPU 执行的硬件操作，
  2. 为内核 C 代码执行做准备的汇编指令，
  3. 决定如何处理该陷入的 C 函数，
  4. 以及系统调用或设备驱动服务例程。
    虽然三种陷入类型之间的共性表明内核似乎可以用单一的代码路径来处理所有陷入，但事实证明，针对两种不同的情况（来自用户空间的陷入和来自内核空间的陷入）采用独立的代码会更加方便。处理陷入的内核代码（汇编或 C 语言）通常被称为处理程序；处理程序最开始的指令通常是用汇编语言（而非 C 语言）编写的，有时被称为向量。

RISC-V trap machinery RISC-V 陷入机制

RISC-V 陷阱处理硬件寄存器

关键控制寄存器（只能在监督者模式下访问）：

  • stvec：内核写入陷阱处理程序地址，发生陷阱时 CPU 跳转到此地址
  • sepc：陷阱发生时保存程序计数器（PC），sret 指令将其复制回 PC
  • scause：存储陷阱原因代码
  • sscratch：内核陷阱处理程序使用的临时寄存器，用于保护用户寄存器
  • sstatus：包含重要控制位
    • SIE：设备中断使能位
    • SPP：指示陷阱来源模式（用户/监督者），控制 sret 的返回模式

陷阱处理硬件自动流程

当发生陷阱时，RISC-V 硬件自动执行以下步骤：

  1. 中断检查：如果是设备中断且 SIE 位为 0，则忽略后续步骤
  2. 禁用中断：清除 sstatus 中的 SIE 位
  3. 保存 PC：将当前 PC 复制到 sepc
  4. 保存模式：将当前模式保存到 sstatus 的 SPP 位
  5. 设置原因：将陷阱原因代码写入 scause
  6. 切换模式：设置为监督者模式
  7. 跳转处理：将 stvec 的值复制到 PC
  8. 开始执行：在新的 PC 地址开始执行陷阱处理程序

多核特性

  • 多核芯片上每个 CPU 都有自己独立的这套寄存器
  • 多个 CPU 可能同时处理不同的陷阱

这个机制确保了陷阱处理的原子性和隔离性，为操作系统提供了可靠的异常处理基础。

CPU不切换到内核页表，不切换内核栈，不保存除pc外的其它寄存器，这些工作交给内核处理。这样设计的理由在于灵活性，内核可以定制陷阱处理。

Traps from user space 从用户空间陷入

陷入的完整路径为 uservec –> usertrap –> userret

xv6处理用户空间陷阱的设计面临一个关键约束：RISC-V硬件在触发陷阱时不会自动切换页表。因此，陷阱处理程序地址在stvec中必须在用户页表中有有效映射，同时内核页表也必须映射相同的处理程序地址，以便在切换到内核页表后能继续执行。

xv6通过使用蹦跳页面解决这一问题。该页面包含uservec陷阱处理代码，映射在每个进程用户页表的固定虚拟地址TRAMPOLINE处，同时在内核页表中也映射到相同地址。这样，陷阱可以在用户页表下开始执行，并在切换到内核页表后继续执行。

uservec开始执行时，所有32个用户寄存器值需要保存。通过sscratch寄存器临时保存a0，使得uservec有一个可用寄存器。内核为每个进程分配一页内存作为trapframe结构，用于保存用户寄存器。该结构映射在用户地址空间的TRAPFRAME地址处。uservec将用户寄存器保存到trapframe后，从trapframe中检索内核栈地址、CPU标识、usertrap函数地址和内核页表地址等信息，切换satp到内核页表，并跳转到usertrap。

usertrap负责确定陷阱原因并处理。它先设置stvec指向内核陷阱处理程序，保存用户程序计数器sepc。根据陷阱类型调用相应处理函数：系统调用调用syscall，设备中断调用devintr，页错误调用vmfault，其他异常则终止进程。系统调用处理会将sepc加4，使返回时执行ecall后的指令。最后，usertrap调用prepare_return设置控制寄存器，准备返回用户空间，并跳转到userret。

userret切换satp回用户页表。由于蹦跳页面在用户和内核页表中映射相同地址，切换后能继续执行。userret从trapframe恢复用户寄存器，包括a0，最后执行sret返回用户空间。

uservec和userret用汇编语言编写，因需要直接处理寄存器保存恢复和页表切换。

Code: System call arguments 代码：系统调用参数

系统调用参数最初存储在用户寄存器中，随后由内核陷阱处理代码（trap code）移动到陷阱帧（trap frame）中。内核函数 argint、argaddr 和 argfd 分别以整数、指针或文件描述符的形式，从陷阱帧中检索第 n 个系统调用参数

某些系统调用将指针作为参数传递，内核必须使用这些指针来读取或写入用户内存。例如，write 系统调用向内核传递一个指向待写入数据的用户态指针。此类指针带来了两项挑战：

  • 安全性与合法性：用户程序可能存在漏洞或具有恶意，它可能向内核传递一个非法指针，或者是一个旨在欺骗内核访问内核内存空间而非用户内存空间的指针。
  • 页表隔离：xv6 内核页表的映射关系与用户页表不同，因此内核无法使用普通的加载（load）或存储（store）指令直接对用户提供的地址进行操作。

内核实现了专门的函数，用于在内核与用户提供的地址之间安全地传输数据。fetchstr 就是一个例子（3624）。诸如 exec 之类的文件系统调用使用 fetchstr 从用户空间检索字符串格式的文件名参数。fetchstr 内部调用 copyinstr 来完成核心工作。

copyinstr 从用户页表 pagetable 中的虚拟地址 srcva 复制最多 max 字节到目标地址 dst。由于该用户页表并非当前的活动页表，copyinstr 使用 walkaddr（其内部调用 walk）在 pagetable 中查找 srcva，从而得到物理地址 pa0。内核页表将所有的物理内存映射到与其物理地址相等的虚拟地址上（等值映射），这使得 copyinstr 能够直接将字符串字节从 pa0 复制到 dst。walkaddr (1520) 会检查用户提供的虚拟地址是否属于该进程的用户地址空间，从而防止程序欺骗内核读取其他内存区域。类似的功能函数 copyout 则负责将数据从内核复制到用户提供的地址。

Traps from kernel space 从内核陷入

kernel/kernelvec.S, kerneltrap() in kernel/trap.c.

Xv6处理内核陷阱的方式与用户陷阱不同。当CPU进入内核后，usertrap将stvec指向kernelvec汇编代码。kernelvec执行时，内核页表和有效内核栈已就绪，它将所有32个寄存器保存到当前被中断内核线程的栈上，然后跳转到C函数kerneltrap。

kerneltrap主要处理设备中断（通过devintr），若非设备中断则视为内核异常并panic。若因定时器中断调用且当前运行的是进程内核线程（非调度器线程），则调用yield让出CPU。工作完成后，kerneltrap恢复之前保存的控制寄存器（如sepc、sstatus）并返回kernelvec，后者从栈恢复所有寄存器并执行sret返回被中断的内核代码。

关键点在于，从用户态进入内核后，usertrap会及时将stvec从uservec更新为kernelvec。在此更新前有一个短暂窗口，此时内核已执行但stvec仍指向用户陷阱处理程序。RISC-V硬件在陷入时自动禁用中断，且usertrap在设置好stvec后才可能重新启用中断，这避免了该窗口期内发生设备中断导致问题。内核陷阱处理依赖于将状态保存在被中断线程的内核栈上，这确保了线程切换时状态的正确保存与恢复。

Page faults 页错误

当使用一个在页表中没有映射的虚拟地址，或映射的PTE_V标志位被清除，或映射的权限位（PTE_R、PTE_W、PTE_X、PTE_U）禁止所尝试的操作时，RISC-V CPU会引发页错误异常
RISC-V区分三种页错误：

  • 加载页错误（由加载指令引起）
  • 存储页错误（由存储指令引起）
  • 指令页错误（由取指操作引起）
    scause寄存器指示页错误的类型，stval寄存器包含无法转换的地址。

页表为内核在虚拟地址和物理地址之间提供了一层间接性，使内核能够控制地址空间的结构和内容。页错误允许内核拦截加载和存储操作，并通过修改页表动态指定这些引用所指向的数据
应用：

  • 写时复制（copy-on-write）的fork机制使内核能够透明地在父子进程间共享内存，避免复制双方均未修改的页面的开销
  • 内存映射文件——内核通过分页机制使文件内容出现在应用地址空间中，并在发生页错误时透明地读取文件块。
  • 惰性内存分配，允许程序申请巨大的虚拟地址空间，但仅在实际读写页面时才支付分配物理内存的成本。
    xv6仅将页错误用于一种目的：惰性分配。

请阅读kernel/sysproc.c中的sys_sbrk()函数和kernel/vm.c中的vmfault函数，并在kernel/trap.c和kernel/vm.c中搜索对vmfault的调用。

Lazy allocation 惰性分配

xv6的惰性分配包含两个部分。
首先，当应用程序通过调用带有SBRK_LAZY标志的sbrk请求内存时，内核会记录地址空间的扩展，但不会分配物理内存，也不会为新的虚拟地址范围创建页表项（PTE）。
其次，当这些新地址发生页错误时，内核会分配一页物理内存并将其映射到页表中

优点:

  • 惰性分配为应用程序提供了便利，因为它们无需精确预测所需的内存大小。如，某个应用程序可能需要处理输入数据，但无法提前预知输入规模。通过惰性分配，应用程序可以按最坏情况申请内存，却无需承担相应的开销：对于应用程序从未使用的页面，内核完全无需进行任何操作。
  • 此外，若应用程序请求大幅扩展地址空间，非惰性分配的sbrk调用代价高昂：如果应用程序申请1GB内存，内核需要分配并清零262,144个4096字节的物理页。惰性分配能将这一成本分摊到时间维度。

缺点

  • 页错误的额外开销，这涉及用户态与内核态的切换

Code 代码

系统调用 sbrk(n) 将进程的内存大小增加（若 n 为负数则缩小）n 字节，并返回新分配区域的起始地址（即原大小）。其内核实现为 sys_sbrk（第3801行）。

若应用程序指定 SBRK_EAGER 标志，该系统调用由 growproc 函数（第2353行）实现。growproc 调用 uvmalloc，而 uvmalloc（第1628行）通过 kalloc 分配物理内存、清零已分配内存，并使用 mappages 将页表项（PTE）添加到用户页表中。

若应用程序以惰性方式分配内存，sys_sbrk 仅将进程的大小（myproc()->sz）增加 n 并返回原大小；它不会分配物理内存或向进程页表添加 PTE。

当进程对缺乏有效页表映射的虚拟地址进行加载或存储时，CPU 将引发页错误异常。usertrap 会检查此情况（第3372行）并调用 vmfault（第1879行）处理页错误。vmfault 检查故障地址是否在之前通过 sbrk 授予的区域内，通过 kalloc 分配一页物理内存、清零该页，并使用 mappages 向用户页表添加 PTE。Xv6 为新页面的 PTE 设置 PTE_W、PTE_R、PTE_U 和 PTE_V 标志。随后，usertrap 在引发故障的指令处恢复进程执行。由于 PTE 现已有效，重新执行的加载或存储指令将无故障运行。

若应用程序使用 sbrk 释放内存，sys_sbrk 会调用 shrinkproc，后者再调用 uvmdealloc。实际工作由 uvmunmap（第1604行）完成，它使用 walk 查找 PTE。由于某些页面可能从未被进程使用因而未被 vmfault 分配，uvmunmap 会跳过未设置 PTE_V 标志的 PTE。若 PTE 映射有效，uvmunmap 调用 kfree 释放其指向的物理内存。

需注意，Xv6 不仅使用进程的页表来指示硬件如何映射用户虚拟地址，还将其作为记录哪些物理内存页分配给该进程的唯一依据。这正是释放用户内存（在 uvmunmap 中）需要检查用户页表的原因。

现实应用：写时复制（COW）fork

许多内核（尽管不包括 xv6）利用页错误实现写时复制（COW）fork。fork 系统调用承诺子进程看到的内存初始内容与父进程在 fork 时的内存相同。一种实现方式是将父进程的全部内存复制到为子进程新分配的物理内存中——这正是 xv6 的做法。复制可能很慢，若子进程能共享父进程的物理内存则效率更高。但直接实现此方式不可行，因为父进程和子进程对共享栈和堆的写入会相互干扰执行。

写时复制 fork 通过恰当使用页表权限和页错误，使父进程与子进程安全共享物理内存。基本方案是让父进程和子进程初始共享所有物理页，但均将其映射为只读（清除 PTE_W 标志）。父进程和子进程随后可从共享物理内存读取。若任一方向共享页写入，RISC-V CPU 将引发页错误异常。支持 COW 的内核会响应此异常：分配一页新物理内存，将共享页内容复制到新页中，修改故障进程页表中相关 PTE 使其指向副本并允许读写，然后在引发故障的指令处恢复故障进程。由于 PTE 现已允许写入，重新执行的存储指令将无故障运行，并修改页的私有副本而非共享页。

写时复制需要簿记管理以确定何时可释放物理页，因为每个页可能被不同数量的页表引用（取决于 fork、页错误、exec 和 exit 的历史）。这种簿记支持一项重要优化：若进程发生存储页错误且该物理页仅被该进程的页表引用，则无需复制。

写时复制使 fork 更快，因为 fork 无需复制内存。部分内存可能在后续写入时仍需复制，但多数内存通常永远无需复制。常见例子是 fork 后接 exec：fork 后可能有少量页被写入，但子进程的 exec 会释放从父进程继承的大部分内存。写时复制 fork 避免了复制这些内存的需要。此外，COW fork 对应用程序是透明的：无需修改即可受益。

现实应用：按需分页

另一项广泛利用页错误的功能是按需分页。在 exec 系统调用中，xv6 在启动应用程序前将其全部文本和数据加载到内存中。由于应用程序可能很大且从磁盘读取耗时，此启动成本对用户可能显著。为减少启动时间，现代内核初始并不将可执行文件加载到内存，而是创建用户页表并将所有 PTE 标记为无效。内核启动程序运行；每次程序首次使用某页时，会发生页错误，内核响应并从磁盘读取该页内容映射到用户地址空间。与 COW fork 和惰性分配类似，内核可透明地实现此功能。

计算机上运行的程序可能需要比 RAM 更多的内存。为优雅处理，操作系统可实现磁盘分页。思路是仅将部分用户页存储在 RAM 中，其余存储在磁盘的分页区。内核将对应存储在分页区（而非 RAM）内存的 PTE 标记为无效。若应用程序尝试使用已被换出到磁盘的页，将引发页错误，该页必须被换入：内核陷阱处理程序将分配一页物理 RAM，从磁盘读取该页到 RAM，并修改相关 PTE 指向 RAM。

若需换入页但无空闲物理 RAM 会发生什么？此时内核必须通过换出或驱逐某物理页到磁盘分页区来释放它，并将指向该物理页的 PTE 标记为无效。驱逐代价高昂，因此若分页不频繁则性能最佳：若应用程序仅使用其内存页的子集且这些子集的并集能放入 RAM。此特性常被称为具有良好的引用局部性。与许多虚拟内存技术一样，内核通常以对应用程序透明的方式实现磁盘分页。

计算机常以极少或无空闲物理内存运行，无论硬件提供多少 RAM。例如，云提供商在单台机器上多路复用多用户以经济高效使用硬件。又如，用户在智能手机的少量物理内存上运行多应用。在此类环境下分配一页可能需先驱逐现有页。因此，当空闲物理内存稀缺时，分配成本高昂。

当空闲内存稀缺且程序仅主动使用部分分配内存时，惰性分配和按需分页尤其有利。这些技术还能避免页面被分配或加载后从未使用或在使用前被驱逐所造成的浪费。

现实应用：内存映射文件

其他结合分页和页错误异常的功能包括自动扩展栈和内存映射文件。内存映射文件是程序通过 mmap 系统调用映射到其地址空间中的文件，使程序能通过加载和存储指令读写它们。
